# MSA — 서비스별 DB 구성과 서비스 간 데이터 전달

본 문서는 마이크로서비스 아키텍처에서 **데이터베이스 경계**와 **서비스 간 데이터 참조 방식**을 팀이 일관되게 적용하기 위한 설명서다. 필수 규칙 요약은 `.cursor/rules/msa-db-and-integration-pointer.mdc`를 따른다.

---

## 1. 목적

- 각 서비스가 **자신의 데이터에 대한 단일 소유권**을 갖도록 한다.
- 서비스 간 결합을 **HTTP API·메시지(이벤트)** 로만 허용하고, **DB 직접 공유**로 인한 은밀한 결합을 막는다.
- API 동기 호출과 RabbitMQ 비동기 전달 중 **목적·성능·복잡도**를 고려해 선택할 수 있도록 가이드를 둔다.

---

## 2. 서비스별 DB 구성 원칙

### 2.1 물리적 분리

- MSA에서는 **서비스마다 전용 PostgreSQL 인스턴스(또는 전용 클러스터/호스트)** 로 **물리적으로 분리**하는 것을 원칙으로 한다.
- 예: `identity-service`는 `identity_db` 전용 인스턴스, `usage-service`는 `usage_db` 전용 인스턴스처럼 **서로 다른 PostgreSQL**에서 동작한다.
- Docker 로컬 환경에서도 **서비스별로 별도의 PostgreSQL 컨테이너(또는 별도 데이터 볼륨·별도 포트)** 로 운영하는 방식과 정합되게 둔다.

### 2.2 한 서비스가 여러 DB가 필요한 경우

- **하나의 서비스**가 여러 데이터베이스 스키마/역할을 써야 한다면, **동일한 PostgreSQL 인스턴스 안에 논리 DB(데이터베이스/스키마)를 여러 개** 두는 방식으로 한정한다.
- 이때도 **다른 서비스의 테이블과는 경계를 섞지 않는다.**

### 2.3 서로 다른 서비스가 각자 DB를 가질 때

- 서비스 A와 서비스 B가 **각자의 전용 PostgreSQL**을 쓰고, 그 안에 **각 서비스 전용 DB(논리 DB)** 를 **독립적으로** 만든다.
- 한 인스턴스에 여러 서비스의 “소유 테이블”을 섞어 두지 않는다.

---

## 3. 타 서비스 DB에 대한 직접 액세스 금지

- 마이크로서비스 아키텍처에서 **어떤 서비스도 다른 서비스가 소유한 데이터베이스에 JDBC 등으로 직접 연결하거나 쿼리해서는 안 된다.**
- 데이터가 필요하면 다음만 허용한다.
  - **동기:** 해당 소유 서비스가 제공하는 **공개·내부 HTTP API** 로 조회·명령
  - **비동기:** 계약된 이벤트를 **RabbitMQ** 로 발행·구독하여 로컬 DB에 **복제(프로젝션)·상태 반영**

---

## 4. 서비스 간 “데이터 참조” 방식 선택

다음 두 가지 중 **목적, 성능, 운영 복잡도, 일관성 요구**를 고려해 선택한다.

| 관점 | HTTP API | RabbitMQ(이벤트) |
|------|----------|------------------|
| 적합한 경우 | 요청 시점에 즉시 필요한 값, 조회 위주, 강한 일관성이 필요한 읽기 | 다른 서비스 로컬 상태를 맞추기, 비동기 처리, 피크 흡수, 장애 시 버퍼링 |
| 유의점 | 호출 횟수·페이로드 크기에 따른 지연·부하 | 소비 지연, 중복, 멱등성, 스키마 진화 |

**성능:** “네트워크가 빠르다”는 이유만으로 **건수만큼 API를 반복 호출**하는 설계는 피한다. 아래 5절을 참고한다.

---

## 5. HTTP API로 다른 서비스 데이터를 참조할 때 (성능)

서비스들은 보통 안정적인 네트워크로 연결되어 **소수의 추가 API 호출**은 체감이 작을 수 있다. 그러나 **조회 건수에 비례해 API 호출이 늘어나는 패턴**은 처리 속도를 크게 떨어뜨리고, 공통 서비스에 부하를 몰아 시스템 장애로 번질 수 있다.

### 5.1 나쁜 예: 건당 다중 호출 (N+1 유사)

가정: 상담 이력 1건을 그릴 때마다 **상담 이력·고객·사용자·부서·상담 유형·상담 코드** 등 부가 정보를 위해 **건당 6번** 다른 서비스(공통 서비스)를 호출한다.

- 상담 이력 **100건** 조회 시 → 공통 서비스에 **600번** API 호출
- 사용자 **10명**이 동시에 전체 이력을 조회하면 → **6,000번** 호출에 가까워져 과부하·지연·장애 가능

고객·사용자·부서 정보는 **다른 기능에서도** 자주 쓰이므로, 이런 패턴은 **전역 병목**이 된다.

### 5.2 개선: 배치·일괄 조회 API

- **같은 종류의 부가 정보**는 **한 번의 API로 묶어서** ID 목록(또는 필터)을 넘겨 **일괄 조회**한다.
- 그러면 조회하는 **데이터 건수와 무관하게(또는 훨씬 느리게만 증가하게)** 호출 횟수를 일정하게 유지할 수 있다.
- **준수 가능한 응답 시간**과 **예상치 못한 폭주 부하**를 줄이는 데 유리하다.

---

## 6. RabbitMQ로 데이터를 전달·참조할 때

RabbitMQ는 **메시지 브로커**로, 수신 서비스가 바쁘거나 연결이 끊겨도 **임시 저장(버퍼)** 역할을 할 수 있다.

### 6.1 동기 호출의 한계

서비스마다 서버가 나뉘어 있을 때, **모든 작업을 동기 호출**로 묶으면 리소스·지연이 커지고 **장애 전파**가 쉽다.

예: “임시저장”에서 처음에는 원본만 저장하고, 이후에는 **변경분만** 계산해 저장한다면, 변경분 계산은 데이터가 커질수록 **오래 걸리는 작업**이 될 수 있다. 이를 **동기 체인**으로만 처리하면 사용자 요청 지연이 커지고, 한 단계 장애가 전체로 퍼진다.

### 6.2 메시지 큐의 역할

- “나중에 처리해도 되는 작업”을 큐에 넣고, 소비 서비스가 **비동기**로 처리하게 두면 **응답 시간을 짧게** 유지할 수 있다.
- 서버 간 **결합(coupling)을 낮추고**, 워커 수를 늘리는 등 **확장**에 유리하다.

### 6.3 구현 시 유의

- **중복 전달·재시도**를 전제로 **멱등성**(같은 이벤트 여러 번 처리해도 결과 일치)을 설계한다.
- 이벤트 스키마 변경은 **버전·호환 정책**과 함께 관리한다.
- “참조 데이터 최신성”이 즉시 필요하면 API 조회와 조합하거나, **로컬 캐시·프로젝션** 전략을 문서화한다.

## 7. 루트 `docker-compose.yml`의 PostgreSQL(로컬)

| Compose 서비스 | 용도 |
|----------------|------|
| `postgres` | **identity-service** 등 — `POSTGRES_DB`(기본 `app`). `team_db` 는 두지 않는다. |
| `postgres-team` | **team-service** 전용 인스턴스·볼륨; `docker/postgres/init/postgres-team-init.sql` 로 `team_db`·`team_app` 생성(psql `-f`, Windows CRLF 회피). |
| `postgres-usage` | **usage-service** 전용 — `02-create-usage-db.sh` 로 `usage_db` 생성. |

### 7.1 team-service 호스트 실행(Windows PowerShell) 예시

`team-service`를 호스트에서 `bootRun` 할 때는 `TEAM_POSTGRES_*`를 명시해 `postgres-team`(기본 `localhost:5434`)에 연결한다.

```powershell
# 팀 서비스 포트
$env:TEAM_SERVICE_PORT = "8094"

# team-service 전용 DB (분리된 postgres-team)
$env:TEAM_POSTGRES_HOST = "localhost"
$env:TEAM_POSTGRES_PORT = "5434"
$env:TEAM_POSTGRES_DB = "team_db"
$env:TEAM_POSTGRES_USER = "team_app"
$env:TEAM_POSTGRES_PASSWORD = "team_app"

# RabbitMQ
$env:SPRING_RABBITMQ_HOST = "localhost"
$env:SPRING_RABBITMQ_PORT = "5672"
$env:SPRING_RABBITMQ_USERNAME = "guest"
$env:SPRING_RABBITMQ_PASSWORD = "guest"

.\gradlew.bat --no-daemon bootRun
```

주의:

- `RABBITMQ_USER`/`RABBITMQ_PASSWORD`가 아니라 **`SPRING_RABBITMQ_USERNAME`/`SPRING_RABBITMQ_PASSWORD`**를 사용한다.
- 기존 세션에 `SPRING_DATASOURCE_*`가 남아 있으면 team 설정을 덮어쓸 수 있으므로 제거 후 실행한다.
- `postgres-team` 초기화가 실패한 이력이 있으면 `team_app` 역할이 없을 수 있다. 이 경우 `postgres_team_data` 볼륨을 재생성한 뒤 다시 기동한다.
